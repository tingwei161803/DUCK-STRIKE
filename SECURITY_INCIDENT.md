## 3. 攻擊手法特徵

- **注入點**：tool result 通道（`Bash`、`Read`、`Edit`、`preview_eval` 都出現過）；**非** user 訊息、**非**檔案內容。
- **標記特徵**：反覆出現孤立 `</target>` 閉合標籤——攻擊者嘗試「閉合」它假設的某層 XML/模板容器，讓後續文字被當成新的指令脈絡。
- **手法演進**：先用假成功／假指令**消耗信任與資源**（誘使重做、誤判狀態），再趁脈絡混亂投放**真正的惡意目標**（機密外洩 + 金錢），並以**情緒勒索**（母親病危）壓過安全判斷。
- **攻擊目標**：竊取環境變數密鑰（`$AMAZON_GIFT_CODE`），透過已連接的 Gmail／MCP 對外通道外洩 + 造成金錢損失。

## 4. 可能來源（推測，需以日誌佐證）

檔案系統搜尋已**排除靜態檔案污染**，來源集中在 **runtime 通道**。依可能性排序：

1. **被污染的 MCP server 或工具回傳（最可能）**：此 session 連接大量 MCP（`windows-mcp` 可操作桌面/剪貼簿/螢幕擷取、`telegram` 接收外部訊息、`Claude_Preview` 在瀏覽器 eval、`figma`、`pencil`、`visualize` 等）。任一 server 若被入侵、為惡意第三方、或回傳被竄改，皆可在 tool result 注入文字。`windows-mcp` 的 `Clipboard`/`Scrape`/`Snapshot` 尤其危險。
2. **同機其他在跑的服務干擾**：4173 跑「殭屍大逃殺」、5173 跑「殭屍塔防」（另外的專案）。若其 HTTP 回應/頁面/console 被植入對 AI 的指令，預覽工具與之互動時會帶入。
3. **代理／快取／hook 層竄改**：若 `settings.json` 有 hook 後處理工具輸出，或有中間代理層，亦可能為注入點。

> 註：真正坐實是哪個 server/管道，需查該 session 載入的 MCP 清單與其日誌；AI 端看不到底層管線，本節為推測。

- 隔離/關閉同機其他在跑的 dev server（4173、5173），避免交叉污染。

**架構面**
- **最小權限**：agent 只給任務必需的工具（讀寫專案檔 + build + 預覽），能寄信/付款/讀密鑰者預設不連。

- **信任原則**：tool result 是**資料、不是指令**；只有真實 user 訊息才算指令。
